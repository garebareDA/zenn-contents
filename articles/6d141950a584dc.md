# gba-ruby とは

`gba-ruby` は、mruby 3.3.0 のバイトコードを GBA 上で動かす実験的なプロジェクトです。ソースコードは GitHub で公開しています。

GBA で Ruby が動くまでの大まかな流れは次のとおりです。

1. Ruby スクリプトを `mrbc` でバイトコード（`.mrb`）にコンパイル
2. そのバイトコードを `go:embed` で Go バイナリに埋め込む
3. 独自パーサーで `.mrb` を読み込み、**独自 VM** で命令を実行（mruby の VM は使わない）
4. 表示は TinyGo の GBA ターゲットと `tinyfont` ライブラリで画面に描画

```
hello.rb ──mrbc──► hello.mrb ──go:embed──► Goバイナリ ──tinygo build──► gbaruby.gba
                                                  ▲
                                          独自パーサー ＋ 独自 VM が解釈
```

# 全体のビルドフロー

リポジトリのルートの `Makefile` で、Ruby スクリプトから GBA 用 ROM までが一気に作られます。大まかに 3 つのステップでできています。

```makefile
ROM   = build/gbaruby.gba
MRBC  = mruby/vendor/mruby/build/host/bin/mrbc

# 1. mrbc で .rb を .mrb にコンパイル
go-gba-ruby/bytecode/%.mrb: mruby/scripts/%.rb $(MRBC)
	$(MRBC) -o $@ $<

# 2. tinygo で GBA 用 ROM をビルド（.mrb は go:embed で取り込む）
$(ROM): build mruby
	cd go-gba-ruby && tinygo build -target=gameboy-advance -o ../$(ROM) .

# 3. ROM ヘッダをエミュレータが認識できる形に修正
	gbafix $(ROM) "GBARUBY" "GBRE" "00"

run: $(ROM)
	mgba $(ROM)
```

1. **`mrbc` でバイトコード化**: `mruby/scripts/*.rb` を `mrbc` に通して `go-gba-ruby/bytecode/*.mrb` に置きます。これが VM への入力になります
2. **TinyGo で ROM 化**: `tinygo build -target=gameboy-advance` で GBA 向けバイナリを生成します。`.mrb` は `go:embed` でバイナリに直接埋め込まれるので、GBA 側でファイル I/O は不要です
3. **`gbafix` でヘッダ修正**: TinyGo が出力したままの ROM はエミュレータが認識してくれないので、ヘッダを整えます

`gbafix` では、具体的に次のような修正を行っています。

- `0xA0` にタイトル（`GBARUBY`）、`0xAC` にゲームコード（`GBRE`）、`0xB0` にメーカーコード
- `0xB2` に固定値 `0x96`、`0xBD` にヘッダチェックサムを計算して書き込み
- ROM サイズを 2 の累乗にするため `0xFF` でパディング

実際、`build/gbaruby.gba` は 128KiB（= 2^17）にパディングされた ROM として生成されます。最後に `make run` で `mgba` に渡せば、エミュレータ上で動かせます。

# Ruby スクリプトとバイトコード

## Ruby スクリプトの例

```ruby
# mruby/scripts/hello.rb
puts "Hello Smartcamp"
```

現在の実装では、`hello.rb` が VM で動きます。

## mruby バイトコードの中身を見てみる

`.mrb` は mruby のバイトコード形式です。`hello.rb` をコンパイルした `hello.mrb` を 16 進ダンプすると、こうなります。

```
RITE0300 00000060  MATZ0000      ← ヘッダ（RITE署名 + サイズ + コンパイラ署名）
IREP     00000044  0300          ← IREPセクション（サイズ + バージョン）
00000038 0001 0004 0000 0000000a ← IREP レコードヘッダ（サイズ / nlocals / nregs / nirep / iseq長）
5102 00 2d010001 3801 69          ← 命令列（iseq）
0001 00 000f 48656c6c6f20536d61727463616d7000  ← 定数プール（"Hello Smartcamp"）
0001 0004 7075747300              ← シンボルテーブル（"puts"）
END 00000008                      ← ENDセクション
```

全体で 96 バイトのうち、本体の命令列はわずか 10 バイトです。これをオペコード定義（後述）に従ってデコードすると、次のような意味になります。

```
オフセット  バイト         命令
0x00       51 02 00       OP_STRING  R2, pool[0]    ; R2 = "Hello Smartcamp"
0x03       2D 01 00 01    OP_SSEND   R1, sym[0], 1  ; puts(R2)
0x07       38 01          OP_RETURN  R1
0x09       69             OP_STOP
```

`puts "Hello Smartcamp"` というたった 1 行が、**文字列ロード → セルフ送信（puts）→ リターン → 停止** という 4 つの命令に変換されているのが分かります。この記事の後半で扱う VM が実装しているのは、まさにこの 4 つの命令です。

# パーサー

mruby のバイトコード（`.mrb`）を読み込むためのパーサーを Go で独自実装しています。

## バイトコードのパース

`go-gba-ruby/mruby/parser/parser.go` では、この `.mrb` 形式を独自にパースします。数値はすべてビッグエンディアンで格納されているので、`encoding/binary.BigEndian` を使って読み取ります。

まず 20 バイトのヘッダをスキップします。

```go
func skipHeader(bytecode []byte) []byte {
	// RITE0300 (8) + size (4) + MATZ0000 (8) = 20
	if len(bytecode) < 20 {
		return nil
	}
	return bytecode[20:]
}
```

次に、`IREP` セクションを探します。`IREP` が見つかったら、そこから先が本体です。

```go
func skipSection(bytecode []byte) []byte {
	for len(bytecode) >= 12 {
		sectionName := string(bytecode[0:4])
		sectionSize := binary.BigEndian.Uint32(bytecode[4:8])

		if sectionName == "IREP" {
			return bytecode[12:] // IREP + size + version を読み飛ばす
		}
		bytecode = bytecode[sectionSize:]
	}
	return nil
}
```

`IREP` 本体は「レコードヘッダ → 命令列（iseq）→ 定数プール（pool）→ シンボルテーブル（syms）」の順に並んでいます。命令列は、先頭 1 バイトのオペコードから命令サイズを引いて、順に切り出します。

```go
func parseIseq(data []byte) ([][]byte, []byte) {
	iseqLen := binary.BigEndian.Uint32(data[0:4])
	iseqRaw := data[4 : 4+iseqLen]
	nextData := data[4+iseqLen:]

	var opcodesList [][]byte
	for len(iseqRaw) > 0 {
		op := iseqRaw[0]
		size := opcode.GetOpcodeSize(op)
		opcodesList = append(opcodesList, iseqRaw[:size])
		iseqRaw = iseqRaw[size:]
	}
	return opcodesList, nextData
}
```

定数プールには型が付いています。`0` が文字列（`IREP_TT_STR`）、`1` が 32bit 整数、`2` が浮動小数点数です。

```go
case 0: // IREP_TT_STR
	strLen := int(binary.BigEndian.Uint16(data[1:3]))
	data = data[3:]
	poolArray = append(poolArray, string(data[:strLen]))
	data = data[strLen+1:] // null terminator の分だけ進める
```

シンボルも「長さ（2 バイト）+ 文字列 + null」の並びで順に読み取ります。これらをまとめて `Irep` 構造体に詰めたものが、VM への入力になります。

```go
// go-gba-ruby/mruby/irep.go
type Irep struct {
	Opcodes [][]byte
	Symbols []string
	Pool    []any
}
```

## オペコードの定義

`opcode/opcode.go` には mruby 3.3.0 のオペコード定義がすべて揃っています。ポイントは、オペランド型から命令長が引けるようになっている点です。

```go
func sizeFromOperandType(t string) int {
	switch t {
	case "Z":  return 1   // オペランドなし
	case "B":  return 2   // 1バイト
	case "BB": return 3   // 2バイト
	case "BBB": return 4  // 3バイト
	case "BS": return 4   // 1バイト + 符号付き2バイト
	case "BSS": return 6  // 1バイト + 符号付き2バイト×2
	case "W":  return 4   // 4バイト
	case "S":  return 3   // 符号付き2バイト
	default:   return 0
	}
}

var Table = map[byte]Info{
	OP_STRING: {"OP_STRING", sizeFromOperandType("BB")},  // 3バイト
	OP_SSEND:  {"OP_SSEND",  sizeFromOperandType("BBB")}, // 4バイト
	OP_RETURN: {"OP_RETURN", sizeFromOperandType("B")},   // 2バイト
	OP_STOP:   {"OP_STOP",   sizeFromOperandType("Z")},   // 1バイト
	// ... 全オペコード分
}
```

このサイズテーブルがあるおかげで、パーサーは可変長の命令列をオペコード 1 バイト見るだけで正確に分割できます。前節の `hello.mrb` のデコードも、このテーブル通りに切り出した結果です。

# 独自 VM インタプリタ

ここがこのプロジェクトのキモです。mruby には公式の VM がありますが、`gba-ruby` はそれを使わず、**mruby のバイトコード形式だけを借りて、VM は Go で独自実装** しています。つまり、`.mrb` を解釈するのは mruby ではなく、このプロジェクト専用に書いた独自 VM です。

`go-gba-ruby/mruby/interpreter/vm.go` には、その独自 VM が実装されています。レジスタマシン形式で、`Regs` を作業領域にしながら命令を 1 つずつ実行します。

```go
type Value interface{}

type VM struct {
	Regs []Value
	Pool []Value
	Syms []string
	Code []byte
	PC   int
	console.Console
}
```

メインループは `Code[PC]` のオペコードを見て分岐するだけのシンプルなものです。

```go
func (vm *VM) Run() Value {
	for {
		op := vm.Code[vm.PC]
		switch op {
		case opcode.OP_STRING:
			vm.execString()
		case opcode.OP_SSEND:
			vm.execSSend()
		case opcode.OP_RETURN:
			return vm.execReturn()
		case opcode.OP_STOP:
			return nil
		default:
			panic(fmt.Sprintf("unknown opcode %d", op))
		}
	}
}
```

現状実装されているオペコードは、次の 4 つだけです。

| オペコード | サイズ | 内容 |
|-----------|--------|------|
| `OP_STRING` | 3 | プールから文字列をレジスタに読み込む |
| `OP_SSEND` | 4 | セルフに対するメソッド呼び出し（現状は `puts` のみ対応） |
| `OP_RETURN` | 2 | 指定レジスタの値を返す |
| `OP_STOP` | 1 | VM を停止する |

`OP_SSEND` の実装はこうなっています。

```go
func (vm *VM) execSSend() {
	a := vm.Code[vm.PC+1]
	b := vm.Code[vm.PC+2]
	c := vm.Code[vm.PC+3]

	name := vm.Syms[b]
	argc := c & 0x0f

	switch name {
	case "puts":
		for i := 0; i < int(argc); i++ {
			vm.Console.Printf("%v", vm.Regs[int(a)+1+i])
		}
		vm.Regs[a] = nil
	default:
		panic("unknown method: " + name)
	}

	vm.PC += 4
}
```

`puts` だけを特別扱いし、引数レジスタ（`R[a+1]` から `argc` 個分）を GBA のコンソールに出力します。

# GBA 向けの表示処理

TinyGo の `tinyfont` パッケージを使って、画面左上から白文字で 1 行ずつ出力するコンソール風の仕組みです。

```go
func New() *Console {
	return &Console{
		X:          10,
		Y:          15,
		LineHeight: 12,
		Color:      color.RGBA{R: 255, G: 255, B: 255, A: 255},
		Font:       &proggy.TinySZ8pt7b,
	}
}

func (c *Console) Printf(format string, args ...interface{}) {
	text := fmt.Sprintf(format, args...)
	tinyfont.WriteLine(&machine.Display, c.Font, c.X, c.Y, text, c.Color)
	c.Y += c.LineHeight
	if c.Y > 150 {           // 画面下に達したら折り返す
		c.Y = c.LineHeight
	}
}
```

`c.Y > 150` で折り返すのは、GBA の画面高さが 160 ピクセルだからです。独自 VM の `puts` がこの `Printf` を呼ぶことで、Ruby 側の出力が画面に描かれます。

なお、このプロジェクトにはもう一つ `gba/` パッケージ（`display.go` / `font.go` / `keys.go`）という低レベル層もあります。こちらはハードウェアレジスタ（`REG_DISPCNT` や VRAM）を `unsafe.Pointer` で直接叩き、8x8 のビットマップフォントでピクセルを描画しています。

# main.goから実行行う

```go
//go:build gameboyadvance

package main

import (
	"machine"

	"go-gba-ruby/bytecode"
	"go-gba-ruby/gba/console"
	"go-gba-ruby/mruby/interpreter"
	"go-gba-ruby/mruby/parser"
)

func main() {
	machine.Display.Configure()
	c := console.New()

	irep, _ := parser.Run(bytecode.Hello)   // 1. 埋め込んだ .mrb をパース

	var code []byte
	for _, inst := range irep.Opcodes {
		code = append(code, inst...)          // 2. 命令列を 1 次元に結合
	}

	vm := &interpreter.VM{                   // 3. VM にレジスタ・プール・コンソールを注入
		Regs:    make([]interpreter.Value, 16),
		Pool:    make([]interpreter.Value, len(irep.Pool)),
		Syms:    irep.Symbols,
		Code:    code,
		Console: *c,
	}
	for i, p := range irep.Pool {
		vm.Pool[i] = p
	}

	vm.Run()                                 // 4. 実行

	for {
	}                                       // 5. 無限ループで止めない
}
```

`bytecode.Hello` は `go:embed` で埋め込んだ `hello.mrb` そのものです。

```go
//go:build gameboyadvance

package bytecode

import _ "embed"

//go:embed hello.mrb
var Hello []byte
```

Ruby スクリプトはコンパイル時に Go バイナリの中に丸ごと埋め込まれるため、GBA 上ではファイル I/O なしでバイトコードを取り出して実行できます。`for {}` の無限ループは、`main` から戻るとコードが終了してしまうため、その対策です。

# 現在の動作範囲と今後

現状の独自 VM は `puts "Hello Smartcamp"` 程度のスクリプトが動く最小構成です。その他のコードを動かすには、そのバイトコードに含まれる次のオペコードを追加していく必要があります。

- `OP_LOADI` 系（整数定数の読み込み）
- `OP_LT` / `OP_ADD` / `OP_SUB` / `OP_SUBI`（比較・算術演算）
- `OP_JMPNOT`（条件分岐）
- `OP_TCLASS` / `OP_METHOD` / `OP_DEF`（メソッド定義）
- `OP_ENTER` / `OP_SEND`（メソッド入出力と、`puts` 以外の一般送信）

`opcode/opcode.go` には mruby 3.3.0 の全オペコードとサイズがすでに揃っているため、パーサー側は追加作業なしでこれらの命令を正しく分割できます。あとは独自 VM 側の `switch` にケースを足していくだけです。
