## 問題: LLMが推測でコードを修正する

TypeScriptのプロジェクトでLLMにコード修正を依頼すると、LLMは型情報を推測して回答することがあります。

```typescript
//極端な例
User: "createUserの引数を追加して"
LLM: 「nameとemailを受け取っていますね。そこにroleを追加します」
// 実際の型は { repo: UserRepository, data: CreateUserRequest } だった…
```

実際は型の書き方が違う、使い方違う、asを使ってしまう等に収まると思いますが。

## 既存の選択肢

TypeScriptのコードを理解する方法として、現在は2つの選択肢があります。

### LSP（Language Server Protocol）

TypeScriptの言語サーバーと通信し、定義・型・参照を正確に取得します。

- ✅ 正確な情報
- ✅ プロジェクトの実態を反映
- ❌ セットアップが面倒
- ❌ 言語ごとにサーバーが必要
- ❌ LLMに組み込むのが重い

### grep

プロジェクトのソースコードを文字列検索します。

- ✅ 簡単
- ✅ どんなプロジェクトでも使える
- ❌ 型情報までは取れない
- ❌ シンボルの意味までは分からない
- ❌ ノイズが多い

### ギャップ

LSPは正確ですが重く、grepは軽いが不正確です。この中間に「**必要十分なコンテキストを手軽に取得する**」選択肢があっても良いのでは、というのがこの記事のモチベーションになっています。

## 提案: TypeScript Context Extractor

TypeScript Context Extractorは、**プロジェクトの型情報を自動収集する軽量なコンテキスト収集器**です。

```
User → Context Extractor → TypeScript Compiler API → 型情報を収集 → LLMへ注入
```

Extractorは以下の責務のみを持っています。

- ユーザー入力の解析
- TypeScriptシンボルの抽出
- TypeScript Compiler APIを使った定義・型の取得
- 取得情報の整理とLLMへの受け渡し

## 実装

### 1. シンボル抽出

ユーザーの日本語入力からTypeScriptのシンボル名を抽出します。

```typescript
// "createUserを修正して" → ["createUser"]
// "Userの型を教えて" → ["User"]
const symbols = extractSymbols(userInput);
```

### 2. プロジェクト存在確認

抽出したシンボルが実際にプロジェクトに存在するかを確認します。「TypeScriptのinterfaceって何？」のような一般質問はスルーします。

```typescript
// getDefinition で実在確認
if (!symbolExistsInProject(symbols)) {
  return; // プロジェクトに関係なし → 何もしない
}
```

### 3. 型情報の自動収集

定義から関連する型を抽出し、自動的に関連型まで取得します。
この型名の抽出は正規表現ベース（`:` / `implements` / `extends` の後に続く大文字始まりの識別子を拾う）の簡易的な実装で、Type Checkerによる厳密な型解決は行っていません。

```
createUser の定義 → UserRepository, CreateUserRequest を発見
  → UserRepository の定義も取得
  → CreateUserRequest の型も取得
  → depth=2 まで再帰探索
```

### 4. コンテキストの注入

収集した情報をLLMへの入力の先頭に追加します。

```
# TypeScript Context

## createUser
### Definition
- File: user.ts:37-39
function createUser(repo: UserRepository, input: CreateUserInput): Promise<User>

### Types
#### CreateUserInput (interface)
export interface CreateUserInput {
  name: string;
  email: string;
  role?: 'admin' | 'member';
}

質問: createUserを修正して
```

LLMはこの情報を「見せられた上で」回答するため、推測する必要がなくなります。

## LSPとの比較

| 項目 | LSP | TypeScript Context Extractor |
|---|---|---|
| 正確性 | ✅ 正確 | △ 正確 |
| セットアップ | ❌ サーバー起動が必要 | ✅ tsconfigがあれば即動く |
| 取得範囲 | 任意（クライアント次第） | ✅ 自動（定義＋関連型） |
| 言語依存 | ❌ 言語ごとに別サーバー | ✅ TypeScriptのみ |
| LLM統合 | ❌ 自前実装が必要 | ✅ CLIでパイプ可能 |

Context ExtractorはLSPの代替ではなく、「LLMがプロジェクトの型を知る」というユースケースに特化した軽量な選択肢です。

## 実際の動作

```bash
# TypeScriptの質問 → 型情報が注入される
$ node cli.js "createUserを修正して"
# TypeScript Context
## createUser
### Definition
function createUser(repo: UserRepository, input: CreateUserInput): Promise<User>
### Types
#### CreateUserInput (interface)
export interface CreateUserInput { name: string; email: string; role?: 'admin' | 'member'; }

質問: createUserを修正して

# 関係ない質問 → 素通り
$ node cli.js "今日の天気は？"
今日の天気は？
```
Context Extractorはコード生成や設計判断を一切行いません。Read と Execute を分離しています。
Extractorは副作用を持たない情報取得コンポーネントとして設計することで、LLMと独立してテスト・改善できます。

## まとめ

- LLMは型情報を推測する。それは仕方ない。
- LSPは正確だが重く、grepは軽いが不正確。
- TypeScript Context Extractor はその中間。
- プロジェクトの実在シンボルだけを狙って、定義＋関連型を自動収集する。

実装: https://github.com/garebareDA/typescript-context-extractor
